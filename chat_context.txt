Project knowledge:
- In OMugs C++, DEBUGIT is a macro in Config.h; Go's DEBUGIT(level int) function is a language-driven replacement for that macro and should not be treated as a mismatch.
- VERSION is expected to differ between OMugs C++ and HolyQuestGo and should not be treated as a Config alignment failure.
- CharCodeMap[128] in C++ Config.h is used by the Osi scanner/token code and should not be treated as a required Config.go alignment item while Osi is out of project scope.
- Buf[256] in C++ Config.h is a reusable temporary formatting buffer; Go's Buf string serves the same role and should be treated as a language-driven type adaptation, not a Config alignment failure.
- ChgDir is an original BigDog.cpp utility function and should not be treated as a Go-only addition.
- C++ ChgDir wraps fs::current_path(...), while Go ChgDir wraps os.Chdir(...); this is a library/API adaptation, not a semantic mismatch.
- In BigDog startup, Go's GetHomeDir() call exists to supply HomeDir before ChgDir(HomeDir); this should be treated as a startup adaptation, not a BigDog alignment failure.
- Rename is an original BigDog.cpp utility function and should not be treated as a Go-only addition.
- Remove is an original BigDog.cpp utility function and should not be treated as a Go-only addition.
- StrInsert is an original BigDog.cpp utility function and should not be treated as a Go-only addition.
- StrVectorGetAt is an original BigDog.cpp utility function and should not be treated as a Go-only addition.
- StrToInt is a Go-side helper created to replace widespread C++ stoi(...) usage and should be treated as an intentional adaptation, not a mismatch.
- WhoIsOnline is intentionally removed from this project scope and should not be treated as a BigDog alignment failure.
- Go's DnodeCount emulates the behavior of C++ Dnode::Count and should not be treated as a semantic alignment failure.
- C++ explicit random seeding with srand(...) does not require a direct equivalent in modern Go and should not be treated as a BigDog alignment failure.

Persistent instructions:
Go Code Style Rules
- All functions are free, no methods
- Do not use defer
- Do not change the case of anything
- Preserve intentional alignment and visual spacing
- Allow normal compact expression formatting where spacing is not serving structure or readability
- Always indent 2 spaces
- Never use TABS, except in formatter-controlled import blocks if the IDE reintroduces them
- NEVER EVER RUN gofmt
- Keep multi-variable var declarations on separate lines (e.g., var x int then var y int), not grouped in a single var (x int, y int) block.
- In consecutive single-line var declarations, pad with spaces so the type columns line up when practical (e.g., var Foo int then var BarBaz string).

C++ to Go Conversion Rules
- Strive for line by line conversion
- Always preserve original function names and case
- If converting class methods to free functions would cause a package-level name collision, use the smallest clear prefix needed to avoid the collision (for example, OpenRoomFile, CloseRoomFile)
- Always preserve original variable names and case
- Convert any multi-line "flower box" function comment into a single // comment directly above the corresponding func line
- Preserve existing comments from the C++ source during conversion
- Do not omit, collapse, or silently rewrite C++ comments unless direct preservation would no longer make sense in Go
- If a comment must be adapted, keep its original intent, placement, and level of specificity as closely as possible so it still accurately describes the Go code
- After converting a function, verify that each meaningful C++ comment still appears in the Go version, either verbatim or appropriately adapted
- Avoid creating new variables; an apparently undefined identifier may be a global
- If a new variable is unavoidable, name it using PascalCase
- Do not move the Close for FileExist
- Pattern: pDnodeActor->pPlayer->Save()         becomes: PlayerSave(pDnodeActor.pPlayer)
- Pattern: pDnodeActor->pPlayer->CreatePrompt() becomes: CreatePrompt(pDnodeActor.pPlayer)
- Pattern: pDnodeActor->pPlayer->GetOutput()    becomes: GetOutput(pDnodeActor.pPlayer)
- Pattern: InvCountInt = stoi(InvCountStr)      becomes: InvCountInt = StrToInt(InvCountStr)
- Pattern: _endthread()                         becomes: os.Exit(1)
- If C++ uses overloaded functions, split them into distinct Go function names because Go does not support overloading (for example, StrFind(...) and StrFind(..., Pos) become StrFind(...) and StrFindAfterPos(...))
- When comparing BigDog.cpp to BigDog.go, exclude the forwarding-wrapper section in BigDog.cpp that exists only to simplify calls to other modules.
- After excluding those forwarding wrappers, BigDog.cpp and BigDog.go are broadly aligned.
