Current Go Code Style Rules intent:
- Free functions only, no methods.
- No defer.
- Do not change case.
- Preserve intentional alignment and visual spacing.
- Allow normal compact expression formatting where spacing is not serving structure or readability.
- Always indent 2 spaces.
- Never use tabs, except in formatter-controlled import blocks if the IDE reintroduces them.
- Convert multi-line "flower box" function comments into a single // comment directly above the corresponding func line.
- Keep multi-variable var declarations on separate lines.
- Align types in consecutive single-line var declarations when practical.

Current C++ to Go Conversion Rules intent:
- Strive for line-by-line conversion.
- Preserve original function names and case.
- Preserve original variable names and case.
- Preserve existing comments.
- If direct preservation of comments no longer makes sense after conversion, adapt wording and placement so the comment still accurately describes the Go code.
- Avoid creating new variables; an apparently undefined identifier may be a global.
- If a new variable is unavoidable, name it using PascalCase.
- Do not move the Close for FileExist.
- Preserve project-specific conversion patterns such as:
  - pDnodeActor->pPlayer->Save() => PlayerSave(pDnodeActor.pPlayer)
  - pDnodeActor->pPlayer->CreatePrompt() => CreatePrompt(pDnodeActor.pPlayer)
  - pDnodeActor->pPlayer->GetOutput() => GetOutput(pDnodeActor.pPlayer)
  - stoi(...) => StrToInt(...)
  - _endthread() => os.Exit(1)
